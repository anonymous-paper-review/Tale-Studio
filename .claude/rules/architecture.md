# Architecture — 신기능/리팩토링 공통 판별 규칙

> 형태 원칙: 서술이 아니라 **판별 질문**. 파일 목록·모델명·라인 참조 금지(부패 방지).
> 다른 rule에 있는 제약은 중복하지 않고 포인터만.

## 0. 제1원칙: 상태 우선

새 기능 설계는 "무엇을 만들까"가 아니라 **"새 진실(state)이 생기는가, 어디 사는가"**에서 시작한다.

- 진실은 한 곳에만. **파생 가능한 값은 저장하지 않는다**
  (예: "생성 중"은 별도 플래그가 아니라 queued 잡의 존재로 도출).
- 두 소비자가 같은 사실을 알아야 하면, 서로 통지하게 만들지 말고 **둘 다 진실을 읽게 한다(pull)**.
  push는 제품 레이어의 표시 결정일 뿐, 상태 동기화 수단이 아니다.

## 1. 신기능 6질문 (순서대로)

1. **상태** — 새 진실인가? DB인가 클라 임시인가? 기존 테이블 확장인가?
2. **성숙도** — 이 복잡도가 현재 제품 단계에 정합한가?
   → active change의 proposal에 상태 소유권·복잡도·트레이드오프를 기록한다.
3. **지능** — 모델이 개입하나? 개입하면 제안까지인가? (§3)
4. **시간** — 요청-응답인가 백그라운드인가? (백그라운드면 기존 잡 패턴 준수:
   submit→webhook→CAS 터미널 전이 — `src/lib/generation-jobs.ts`와 generate-* 라우트가 진실)
5. **비밀/과금** — 키·과금·권한이 닿는가? → 무조건 서버 (route+lib)
6. **도메인** — 어느 스테이지 소속인가? → 해당 `features/`·store·`api/` 칸

## 2. 책임 그리드 — 칸별 입주 자격 (판별 질문)

| 칸 | 입주 질문 | 금지 |
|---|---|---|
| 컴포넌트 | "디자이너가 바꿀 수 있는 결정인가?" | 비즈니스 규칙, 직접 fetch |
| store | "컴포넌트가 unmount돼도 기억해야 하나?" | 비밀·서버 로직, persist에 비직렬화 값 |
| route | "클라에 노출되면 사고인가?" — 얇게: 인증·zod 검증·쿼터·위임 | 도메인 로직 비대, SDK 직접 호출(wrapper 경유) |
| lib | "여러 route/store가 쓰나?" | UI 의존 |
| DB | "재시작에도 살아야 하나? 다른 소비자도 읽나?" | 파생값 저장 (§0) |

## 3. 모델 레이어 (⊂ lib) vs 제품 레이어

- 판별: **"LLM 호출이 실패해도 의미 있는 코드인가?"** → yes면 제품 레이어.
  모델 레이어 = 호출 1회를 만드는 것 전부 (컨텍스트 조립·프롬프트·호출·출력 파싱).
- **모델은 제안만 한다.** 실행·검증·과금 방어는 항상 제품 레이어
  (화이트리스트 검증 → 명시적 apply). 프롬프트의 가드는 보조 방어일 뿐 최종 방어가 아니다.
- 모델의 인지는 **pull** — 요청 시점에 진실(DB)을 직렬화해 주입.
  이벤트는 새 호출을 깨우는 트리거가 될 수 있으나, 깨워진 호출도 컨텍스트는 pull로 조립한다.

## 4. 금지 화살표 (설계 냄새)

- 컴포넌트 → DB/외부 API 직접
- store → store import (`project-store`만 예외 — 크로스스테이지는 lib 경유 `getState()`)
- route 안 도메인 로직 비대 (lib로)
- 같은 진실의 이중 기록 (코드↔문서 포함 — 코드가 진실)
- 모델 출력의 무검증 실행

## 5. 원천/파생 — 다단계 데이터 정합 (§0의 연장, decisions #57)

같은 데이터가 여러 스테이지에서 읽히고 고쳐질 때의 판별 규칙.

- **제1판별**: "사람이 직접 정한 값인가(원천), 무언가를 읽어 생성된 결과물인가(파생)?"
  파생물이면 sync를 설계하지 말고 **무엇으로 만들었는지(provenance, 입력 지문)**를 기록한다.
- **5원칙**:
  1. **빌드는 독립** — 각 스테이지의 생성물은 다른 스테이지의 생성물을 절대 고치지 않는다.
  2. **원천은 공동 편집** — 자율 실행(파이프라인)은 **빈칸만** 채운다. 차 있는 칸의 덮어쓰기는
     사람의 명시적 행동(폼·채팅·명시 요청)만. 보호 플래그가 아니라 쓰기 경로의 시멘틱으로.
  3. **합류는 하류** — 스테이지 간 영향은 옆(통지/sync)이 아니라 아래로만:
     확정·선택된 산출물이 하류 빌드의 *재료*가 된다. 역류는 데이터가 아니라 사람의 새 결정.
  4. **일관성 = 순간이 아니라 수습 가능성** — 전역 불일치 상태는 정상. 조건 셋:
     파생물이 입력 지문을 알고 / 낡음(stale)이 사용자에게 보이고 / 명시적 재생성으로 수렴 가능.
  5. **통합 경험은 에이전트** — 스테이지 연결은 데이터 계층의 자동 연쇄가 아니라
     에이전트가 낡음을 읽고 재생성을 *제안*하는 것. 자동화하지 않은 빈자리가 에이전트의 일자리.
- **금지**: 상류 변경 시 하류 자동 무효화·자동 재생성 (사람의 선별 작업 파괴 + 과금).
  stale은 정보, 행동은 항상 명시적.
- **초기생성 vs 재생성을 구분해 설계하지 않는다** — 같은 행위. 유일한 구분은
  "빈칸 채우기(자율 가능 — **과금 파생물 포함**) vs 차 있는 것 교체(사람만)".
  빈칸 자율 채움: 트리거 무관(파이프라인·화면 진입), **상류가 확정한 입력**으로 + **멱등(차 있으면 skip)**
  + **진입/세션당 1회** + **실패는 배지**(과금·재시도 루프 방지). 비용이 크면(예: 다뷰 턴어라운드)
  옵트인으로 좁히는 건 비용 튜닝이지 원칙이 아니다.
  (2026-06-12 amend: 옛 "과금 파생물은 빈칸이라도 사람 방아쇠" 예외 폐기 — 사람 전속은 *차 있는 것 교체*뿐.)
- **보존 판별**: "갈아엎으면 잃는 게 있는가(사람이 그 특정 결과물을 선별·편집했는가)?"
  없으면 통째 재생성이 곧 정합성 메커니즘(신선도 추적 불필요). 사람 손이 닿는 순간 보존 티어로 승격.
- 입력 지문은 **생성 입력을 조립하는 그 함수**가 함께 계산한다 — 분리하면 지문과 실제 입력이
  어긋나 stale 판정이 거짓말을 한다. 그 함수의 입력 목록이 곧 파생 엣지의 선언.
