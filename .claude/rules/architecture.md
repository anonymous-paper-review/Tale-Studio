# Architecture — 신기능/리팩토링 공통 판별 규칙

> 형태 원칙: 서술이 아니라 **판별 질문**. 파일 목록·모델명·라인 참조 금지(부패 방지).
> 다른 rule에 있는 제약은 중복하지 않고 포인터만.

## 0. 제1원칙: 상태 우선

새 기능 설계는 "무엇을 만들까"가 아니라 **"새 진실(state)이 생기는가, 어디 사는가"**에서 시작한다.

- 진실은 한 곳에만. **파생 가능한 값은 저장하지 않는다**
  (예: "생성 중"은 별도 플래그가 아니라 queued 잡의 존재로 도출).
- 두 소비자가 같은 사실을 알아야 하면, 서로 통지하게 만들지 말고 **둘 다 진실을 읽게 한다(pull)**.
  push는 제품 레이어의 표시 결정일 뿐, 상태 동기화 수단이 아니다.

## 1. 신기능 6질문 (순서대로)

1. **상태** — 새 진실인가? DB인가 클라 임시인가? 기존 테이블 확장인가?
2. **성숙도** — 이 복잡도가 현재 제품 단계에 정합한가?
   → `specs/_DECISION_TEMPLATE.md`를 **게이트로 통과**시킨다 (기록 양식 아님 —
   채운 내용은 작업 메모리, 영속은 proposal 한 섹션 또는 decisions.md 한 줄 압축만).
3. **지능** — 모델이 개입하나? 개입하면 제안까지인가? (§3)
4. **시간** — 요청-응답인가 백그라운드인가? (백그라운드면 기존 잡 패턴 준수:
   submit→webhook→CAS 터미널 전이 — `src/lib/generation-jobs.ts`와 generate-* 라우트가 진실)
5. **비밀/과금** — 키·과금·권한이 닿는가? → 무조건 서버 (route+lib)
6. **도메인** — 어느 스테이지 소속인가? → 해당 `features/`·store·`api/` 칸

## 2. 책임 그리드 — 칸별 입주 자격 (판별 질문)

| 칸 | 입주 질문 | 금지 |
|---|---|---|
| 컴포넌트 | "디자이너가 바꿀 수 있는 결정인가?" | 비즈니스 규칙, 직접 fetch |
| store | "컴포넌트가 unmount돼도 기억해야 하나?" | 비밀·서버 로직, persist에 비직렬화 값 |
| route | "클라에 노출되면 사고인가?" — 얇게: 인증·zod 검증·쿼터·위임 | 도메인 로직 비대, SDK 직접 호출(wrapper 경유) |
| lib | "여러 route/store가 쓰나?" | UI 의존 |
| DB | "재시작에도 살아야 하나? 다른 소비자도 읽나?" | 파생값 저장 (§0) |

## 3. 모델 레이어 (⊂ lib) vs 제품 레이어

- 판별: **"LLM 호출이 실패해도 의미 있는 코드인가?"** → yes면 제품 레이어.
  모델 레이어 = 호출 1회를 만드는 것 전부 (컨텍스트 조립·프롬프트·호출·출력 파싱).
- **모델은 제안만 한다.** 실행·검증·과금 방어는 항상 제품 레이어
  (화이트리스트 검증 → 명시적 apply). 프롬프트의 가드는 보조 방어일 뿐 최종 방어가 아니다.
- 모델의 인지는 **pull** — 요청 시점에 진실(DB)을 직렬화해 주입.
  이벤트는 새 호출을 깨우는 트리거가 될 수 있으나, 깨워진 호출도 컨텍스트는 pull로 조립한다.

## 4. 금지 화살표 (설계 냄새)

- 컴포넌트 → DB/외부 API 직접
- store → store import (`project-store`만 예외 — 크로스스테이지는 lib 경유 `getState()`)
- route 안 도메인 로직 비대 (lib로)
- 같은 진실의 이중 기록 (코드↔문서 포함 — 코드가 진실)
- 모델 출력의 무검증 실행
