## 0. 물리적 제약 + 모델 스택

파이프라인의 모든 설계는 이 제약 위에 세워진다.

### 모델 스택 (로컬)

```
┌─────────────────────────────────────────────────────────┐
│ 이미지 생성:    Qwen3 Image                              │
│              ↪ 신규 캐릭터, 신규 로케이션, 첫 프레임 생성  │
│                                                         │
│ 이미지 편집:    Qwen3 Image Edit                         │
│              ↪ 기존 캐릭터의 시간/상태 변화 (의상, 부상,  │
│                 나이, 감정 등)                           │
│                                                         │
│ 영상 생성:      Hunyuan Video 1.5                        │
│              ↪ I2V 전용 (이미지 + 모션 프롬프트 → 5초)    │
└─────────────────────────────────────────────────────────┘
```

### 물리적 제약

```
┌─────────────────────────────────────────────────────────┐
│ 제약 1: 원자 단위 = ~5초                                  │
│   Hunyuan Video 1.5가 안정적으로 출력하는 단위.            │
│   이보다 줄일 수는 있지만 늘리면 품질 급락.                 │
│   모든 영상 = 5초 클립 N개의 조립품.                       │
│                                                         │
│ 제약 2: 프롬프트 = 소프트 리밋                              │
│   영상 생성 모델에 LLM 내장 → 하드 글자 수 제한 없음.       │
│   그러나 길어질수록 성능 저하 → 최적 밀도 스윗 스팟 존재.    │
│                                                         │
│ 제약 3: 모든 샷 = I2V                                     │
│   Hunyuan은 I2V 전용. 모든 샷은 첫 프레임 이미지 필요.       │
│   → 첫 프레임은 Qwen3 Image / Image Edit으로 생성.        │
│                                                         │
│ 제약 4: 캐릭터 영속성은 에셋 시스템으로                       │
│   AI 모델은 같은 캐릭터를 자동으로 일관되게 생성 못 함.       │
│   → 각 캐릭터의 "에셋 상태"를 명시적으로 관리하고             │
│     매 샷마다 적절한 에셋을 사용해 첫 프레임 생성.            │
│                                                         │
│ 제약 5: 샷 액션 복잡도 한계 (가장 중요)                       │
│   5초 안에 너무 많은 행동/사건을 요청하면 영상이 무너짐.        │
│   - 캐릭터 동작이 여러 단계로 끊김                          │
│   - 객체가 사라지거나 모핑                                 │
│   - 카메라 + 캐릭터 + 환경 변화가 동시에 = 혼돈              │
│   → 1샷 = 1 주요 액션 원칙. 보조 액션은 1개까지.             │
│                                                         │
│ 제약 6: API 트렌드 -- start frame 점진적 미지원             │
│   최신 영상 API들은 품질 문제로 start frame I2V를 줄이는 추세. │
│   → Hunyuan은 아직 I2V 지원하지만, 백업 플랜 필요.           │
│   → 첫 프레임 의존도를 낮추는 설계 (모션 프롬프트의 충분성)    │
└─────────────────────────────────────────────────────────┘
```

## 1. 설계 원칙

| 원칙 | 설명 |
|------|------|
| **선형 흐름** | S 먼저, V 나중. 사용자가 한 방향으로 진행 |
| **C 메타레이어** | C는 S 검증 + S→V 변환 규칙 + V 검증으로 두 지점에서 작동 |
| **피드백 보상** | 양방향 협상의 장점을 4가지 메커니즘으로 시뮬레이션 |
| **잠금 정책** | 상위 결정을 함부로 못 바꿈. 바꾸려면 비용 명시 |
| **AI 자동 매핑** | S 결정 → C 규칙 적용 → V 추론/제안 |
| **5초 원자 단위** | 최종 출력은 5초 클립의 시퀀스. 영상 길이 = 샷 개수 |
| **깊이 스케일링** | 영상 길이(샷 수)에 따라 S, C의 활성 범위가 자동 조절 |

### 왜 선형인가

| 이유 | 설명 |
|------|------|
| UX 복잡도 | S와 V 동시 조작 UI = 전문가 전용, B2B 진입장벽 |
| AI 파이프라인 | LLM이 스토리 생성 → 비주얼 추론이 자연스러운 흐름 |
| 의존성 비대칭 | S→V가 V→S보다 압도적으로 강함 (90% vs 10%) |
| 사용자 멘탈 모델 | "이야기 쓰고 → 영상으로 만든다"가 직관적 |

### 선형에서 잃는 것과 보상 전략

| 잃는 것 | 빈도 | 보상 메커니즘 |
|---------|------|-------------|
| V→S 발견 (비주얼이 스토리에 영감) | 10% | Mid Preview + Back Adjust |
| 동시 최적화 (S+V를 같이 보며 탐색) | 프로 수준 | Mid Preview (유일한 양방향 지점) |
| 제약 사전 인식 (V를 모르고 S를 쓰면 실현 불가능) | 30% | Forward Hint |
| 우연한 시너지 | 예측 불가 | Genre Preset |

## 2. 전체 흐름

```
시간축 →

[PRESET]─>[S0]─>[S1]─>[S2]─>[S3]─>[C적용①]─>[MID]─>[L0+L1]─>[L2]─>[C적용②]─>[샷시퀀스]─>[에셋]─>[생성]─>[조립]
(선택적)   │     │     │     │      │         │       │       │      │          │         │       │      │
          hint  hint  hint  hint  S검증    S↔V조감  back    back  S→V변환    샷메타     캐릭터  I2V    최종
                                 인과체크             adj     adj  모티프매핑    데이터    에셋    영상   영상
                                 CDQ확인                      프롬프트생성              + 첫프레임 생성
                                 핍진성                       우선순위정렬              생성    (Hunyuan)
                                 클리셰                                                (Qwen3)
```

생성 단계가 **3단계로 분리**됨:
1. **샷 시퀀스**: 메타데이터 (S/C/V 정보, 프롬프트)
2. **에셋 준비**: Qwen3로 캐릭터 에셋 + 샷 첫 프레임 생성
3. **영상 생성**: Hunyuan으로 첫 프레임 → 5초 영상

### 깊이 레벨 (S0에서 결정)

S0의 러닝타임/포맷이 확정되면 깊이 레벨이 결정되고, 이후 모든 단계의 활성 범위가 조절됨.

| 레벨 | 샷 수 | 러닝타임 | 활성 S | 활성 C | 파이프라인 변형 |
|------|------|---------|-------|-------|-------------|
| **D1 Spark** | 1~2 | ~5~10초 | S0(톤), S3(1씬) | C2(Hook 1개) | [S0]→[S3+Hook]→[V]→생성 (미구현) |
| **D2 Beat** | 3~6 | ~15~30초 | S0, S2약식, S3 | C1(단일인과), C2(미니CDQ), C3(기본템포) | [S0~S3 약식]→[C 약식]→[V]→생성 (미구현) |
| **D3 Arc** | 6~30 | 1~5분 | S0~S3 기본 | C1~C5 기본, C6 | [S0~S3]→[C]→[Mid]→[V]→생성 |
| **D4 Short** | 30~60 | 5~10분 | S0~S3 풀 (서브플롯 0~1) | C1~C6 (가벼움) | 풀 파이프라인 |
| **D5 Featurette** | 60~120 | 10~20분 | S0~S3 풀 + 서브플롯 1~2 | C1~C6 풀 | 풀 파이프라인 |
| **D6 Episode** | 120~180 | 20~30분 | S0~S3 풀 + 서브플롯 2~3 (앙상블) | C1~C6 풀 + 다층 (서브 모티프) | 풀 파이프라인 |
| **D7 Epic** | 180+ | 30분+ | S0~S3 풀 + 서브플롯 다수 | C1~C6 풀 + 에피소드 연속성 | 풀 파이프라인 + 에피소드 연속성 |

> 2026-04-20 개정: 기존 D4 (5~30분)를 D4/D5/D6 3단계로 세분화. 기존 D5 (장편)는 D7로 명명. 코드 구현 범위는 D3~D7.

## 3. Phase별 상세

### Phase 0: Genre Preset (선택적)

S0 + L0 + L1을 동시에 세팅하는 출발점.
Dual-Axis의 "동시 결정" 효과를 프리셋으로 달성.

```
┌───────────────────────────────────────────────┐
│ 프리셋 선택                                    │
│                                               │
│  느와르 미스터리 │ 지브리 판타지 │ 사이버펑크 액션 │
│  키즈 코미디    │ 로맨틱 드라마  │ 다큐 스타일    │
│  호러          │ 뮤직비디오     │ 빈 프로젝트    │
│                                               │
│ 각 프리셋 = S0 기본값 + L0~L1 기본값 + L3 레시피 │
│ 사용자는 이후 자유롭게 수정 가능                  │
└───────────────────────────────────────────────┘
```

**프리셋 라이브러리:**

| 프리셋 | S0 기본 | L1 핵심 | L2 핵심 | L3 레시피 |
|--------|--------|---------|---------|-----------|
| 느와르 미스터리 | 미스터리/어두운 | 각진, 하이콘트라스트 | 틸앤앰버, 아르데코 | 스릴러 |
| 지브리 판타지 | 판타지/따뜻한 | 둥근, 수채화 | 자연색, 유기 환경 | 서사시+드라마 |
| 사이버펑크 액션 | SF+액션/강렬 | 각진+네온 | 네온+다크, 미래도시 | 액션 |
| 키즈 코미디 | 코미디/밝은 | 둥근, 플랫컬러 | 비비드, 깨끗 | 코미디 |
| 로맨틱 드라마 | 드라마/부드러운 | 페인터리 | 워밍, 파스텔 | 로맨스+드라마 |
| 다큐 스타일 | 다큐/사실적 | 포토리얼 | 뮤트/내추럴 | 드라마(미니멀) |
| 호러 | 호러/어두운 | 왜곡, 하이콘트라스트 | 디새추, 어둠 | 호러 |
| 뮤직비디오 | 뮤직/에너지 | 자유/실험적 | 하이채도/네온 | 초현실 |

### Phase 1: Story Foundation (S0 → S3)

순차적으로 S 결정. 각 단계에서 Forward Hint 동반.

**S0: 장르/톤**
```
입력:
  - 장르 (코미디, 드라마, 호러, 액션, SF, 판타지, 혼합...)
  - 톤 (밝은, 어두운, 진지한, 유머러스, 몽환적...)
  - 타겟 감정 (감동, 긴장, 경이, 공포, 웃음...)
  - 러닝타임 / 포맷 (숏폼, 미드폼, 롱폼)

[Forward Hint 표시]
  → 장르별 비주얼 특성 요약
  → AI 영상 생성 제약 사항 (어두운 씬 품질, 빠른 움직임 아티팩트 등)
```

**S1: 내러티브 구조**
```
입력:
  - 구조 유형 (3막, 기승전결, 순환, 비선형)
  - POV (1인칭, 3인칭 제한, 전지적)
  - 주제/모티프 (자유, 고독, 복수, 성장...)
  - 핵심 반전/클라이맥스 위치

[Forward Hint 표시]
  → 씬 수 예측 → 비주얼 에셋 규모
  → POV → 카메라 제약 (1인칭=POV샷 중심)
```

**S2: 캐릭터/관계**
```
입력:
  - 주요 캐릭터 (이름, 역할, 성격, 목표)
  - 캐릭터 아크 (시작 상태 → 끝 상태)
  - 캐릭터 관계맵 (동맹, 적대, 변화)
  - 서브텍스트 (각 캐릭터가 숨기는 것)

[Forward Hint 표시]
  → 캐릭터 수 → 에셋 작업량 예측
  → 성격 → 쉐이프 제안 (착한=둥근, 악한=각진)
  → 아크 → 컬러 아크 방향 (타락=어두워짐, 성장=밝아짐)
```

**S3: 씬 브레이크다운**
```
입력:
  - 씬 목록 (번호, 로케이션, 참여 캐릭터)
  - 씬별 목적 (정보전달 / 감정전환 / 갈등고조 / 해소)
  - 씬별 감정 비트 (시작감정 → 끝감정)
  - 대사 핵심 (키 다이얼로그)
  - 정보 비대칭 (관객 vs 캐릭터가 아는 것)

[Forward Hint 표시]
  → 로케이션 수 → 에셋 규모
  → 감정 범위 → 컬러 스크립트 복잡도 예측
  → 감정 곡선 초안 시각화
```

### Checkpoint: Mid Preview

**S 전체 완료 → V 진입 전. 유일한 양방향 협상 지점.**

```
AI가 생성하는 것:

1. V축 전체 추천 세팅 (L0~L1)
2. 컬러 스크립트 초안 (씬별 컬러/무드 아크)
3. 캐릭터별 비주얼 방향 제안
4. 스토리 감정 곡선 ↔ 비주얼 강도 곡선 매핑
5. AI 생성 난이도 예측
```

```
사용자가 할 수 있는 것:

┌───────────────────────────────────────────────────┐
│                                                   │
│  [V 수락]     → AI 제안 그대로 V축 진행            │
│  [V 수정]     → 팔레트, 스타일 등 직접 조정         │
│  [다른 V 제안] → AI에게 다른 비주얼 방향 요청       │
│  [레퍼런스]   → "블레이드러너 2049 같은 느낌" 입력   │
│  [S 수정]     → 뒤로 돌아가서 스토리 수정           │
│                (S 해당 레벨만 재작업)                │
│                                                   │
└───────────────────────────────────────────────────┘
```

### Phase 2: Visual Foundation (L0 + L1)

Mid Preview 제안 기반. 사용자 조정 가능.

```
L0 확정: 매체, 해상도, fps, 비율, 렌더
L1 확정: 스타일, 쉐이프, 라인, 캐릭터 코어, 텍스처

[Back Adjust]
  L0/L1 작업 중 → S2, S3 수정 가능 (S0, S1은 잠금)
```

### Phase 3: Production Design (L2)

S에서 자동 매핑 + 사용자 수정.

```
자동 생성:
  S2 캐릭터 아크 → L2 컬러 의미 + 의상 아크
  S3 씬 감정 비트 → L2 컬러 스크립트
  S3 로케이션 → L2 로케이션 디자인

[Back Adjust]
  L2 작업 중 → S3 수정 가능 (S0, S1, S2, L0, L1은 잠금)
```

### Phase 4: Shot Execution (L3)

S에서 자동 매핑 + 장르 레시피 적용 + 사용자 미세 조정.

```
자동 생성:
  S3 씬 목적/감정 → L3 장르 레시피 선택
  S3 대사 → L3 카메라 커버리지 패턴
  S3 감정 비트 → L3 조명/그레이딩/음악 매핑

[Back Adjust]
  L3 작업 중 → S3 해당 씬만 수정 가능 (나머지 전부 잠금)
```

## 4. 피드백 메커니즘 상세

### 4.1 Forward Hint

| 대상 | 보여주는 것 |
|------|-----------|
| S0 장르 | L1~L3 장르 레시피 요약 + AI 생성 제약 |
| S0 톤 | L2 팔레트 방향 + L3 조명/편집 기조 |
| S1 구조 | 씬 수 예측 → 에셋 규모 |
| S1 POV | L3 카메라 제약 (1인칭=POV샷) |
| S2 캐릭터 수 | L2 에셋 규모 + L1 디자인 작업량 |
| S2 성격 | L1 쉐이프 제안 |
| S3 로케이션 수 | L2 에셋 규모 |
| S3 감정 범위 | L2 컬러 스크립트 복잡도 |

### 4.2 Mid Preview

S 완료 후 AI가 V 전체를 한 번에 제안하는 체크포인트.
**선형 파이프라인에서 양방향 협상을 시뮬레이션하는 유일한 지점.**

### 4.3 Back Adjust

V 작업 중 하위 S를 제한적으로 수정.

**핵심 규칙: "위로는 못 올라간다"**

```
현재 위치      수정 가능한 S         잠금된 S
─────────     ───────────         ──────────
L0/L1 작업중   S2, S3              S0, S1
L2 작업중      S3                  S0, S1, S2
L3 작업중      S3 해당 씬만         S0, S1, S2 + 다른 씬의 S3
```

잠금 해제가 필요하면 → Mid Preview로 돌아감 (명시적 "리셋").

### 4.4 Genre Preset

프로젝트 시작 시 S+V를 동시에 세팅.
양방향 협상의 "동시 결정" 효과를 프리셋으로 달성.

## 5. 잠금 정책 (Lock Policy)

```
진행 방향 →
                                                            현재 위치
S0 ─→ S1 ─→ S2 ─→ S3 ─→ [MID] ─→ L0 ─→ L1 ─→ L2 ─→ L3 ─→ ●

잠금:
S0  LOCKED ──────────────────────────────────────────────────
S1  LOCKED ──────────────────────────────────────────────────
S2  LOCKED ──────────────────────────────────  수정가능 ─────
S3  LOCKED ──────────────────  수정가능 ─── 해당씬만 ─────────
L0  LOCKED ──────────────────────────────────────────────────
L1  LOCKED ──────────────────────────────────────────────────
L2  LOCKED ──────────────────────────  수정가능 ─────────────
L3  자유 ────────────────────────────────────────────────────
```

**잠금 해제 비용:**

| 돌아가는 곳 | 무효화 | 비용 |
|-----------|--------|------|
| S3 수정 | 해당 씬 L2+L3 | 낮음 |
| S2 수정 | 전체 L2+L3 | 중간 |
| S1 수정 | S2+S3+V 전체 | 높음 (사실상 새 프로젝트) |
| S0 수정 | 전부 | 새 프로젝트 |
| L2 수정 | 해당 요소 L3 | 낮음 |
| L1 수정 | 전체 L2+L3 | 높음 |
| L0 수정 | L1+L2+L3 전체 | 새 프로젝트 |

## 6. S → V 자동 매핑 규칙

### S0 → V

```
장르 ───> L0 포맷 추천 + L1 스타일 방향 + L3 레시피
톤   ───> L2 팔레트 기조 + L3 조명/편집 기조
```

### S2 → V

```
캐릭터 성격 ───> L1 쉐이프 + L2 컬러 배정 + L2 의상 방향
캐릭터 아크 ───> L2 컬러 아크 + L2 의상 아크
  타락: 밝은→중간→어두운
  성장: 뮤트→채도 상승
  비극: 따뜻한→차가운
```

### S3 → V

```
씬 감정 비트 ───> L2 컬러 스크립트 + L3 조명(키:필 비율) + L3 편집(페이싱)
  희망 → 따뜻/밝은, 1:1~2:1, 긴 테이크
  절망 → 차가운/어두운, 8:1+, 느린 페이스
  긴장 → 하이콘트, 4:1, 점점 빠른 컷
  충격 → 스매시컷, 정적→스팅어

씬 목적 ───> L3 카메라 커버리지
  대화 → 마스터 + OTS + CU
  액션 → 트래킹 + 와이드 + 인서트
  감정 → 슬로우 돌리 + ECU + 긴 테이크
```

## 7. 에셋 시스템 + 5초 샷 생성

### 7.1 에셋 시스템 (영속성 관리)

**핵심**: AI 모델은 같은 캐릭터를 자동 일관되게 생성 못 함. 각 캐릭터의 "에셋 상태"를 명시적으로 관리해 매 샷마다 적절한 에셋으로 첫 프레임 생성.

```
에셋 = "특정 시점/상태의 캐릭터/로케이션 이미지"

각 캐릭터는 여러 에셋을 가짐:
  카이 (캐릭터)
    ├─ asset_v1: 깨끗한 트렌치코트, 평온한 표정 (씬1~2)
    ├─ asset_v2: 같은 의상, 충격받은 표정 (씬3 초)
    ├─ asset_v3: 같은 의상, 차가운 표정 (씬3 후~4)
    └─ asset_v4: 비에 젖음, 결의 표정 (씬4 후반)

각 로케이션도 에셋:
  카페 (로케이션)
    ├─ asset_a: 정적인 카페, 비 오는 창
    └─ asset_b: 카메라 위치 다른 같은 카페
```

### 7.2 에셋 생성 흐름 (Qwen3 Image / Image Edit)

```
신규 캐릭터:
  S2 캐릭터 정의 → Qwen3 Image (text-to-image)
    입력: "40대 남성, 검은 트렌치코트, 회색 짧은머리, 진중한 얼굴, 영화적 조명"
    출력: asset_v1 (마스터 에셋)

캐릭터 변화 (의상/감정/상태):
  asset_v1 → Qwen3 Image Edit (image editing)
    입력: asset_v1 + "표정을 충격으로 변경, 같은 외형 유지"
    출력: asset_v2

샷 첫 프레임 생성:
  필요한 에셋들 + 환경 묘사 → Qwen3 Image (composition)
    입력: 캐릭터_에셋 + 로케이션_에셋 + 카메라/조명 프롬프트
    출력: 샷의 첫 프레임 이미지

→ 모든 에셋은 재사용 가능한 "라이브러리"로 저장
```

### 7.3 영상 생성 (Hunyuan Video 1.5)

```
모든 샷 = I2V (Hunyuan은 I2V 전용)

[샷 첫 프레임 이미지] + [모션 프롬프트]
                ↓
        Hunyuan Video 1.5
                ↓
          5초 영상 클립

모션 프롬프트 예시:
  "캐릭터가 천천히 일어선다. 카메라는 정적. 트렌치코트 옷깃이 살짝 흔들린다."
```

### 7.4 두 가지 연속성 레이어

```
레이어 1: 캐릭터/로케이션 영속성 (에셋 시스템이 담당)
  ├─ 같은 캐릭터의 외형/의상이 샷 간 일관
  ├─ 같은 로케이션의 공간 구조가 일관
  └─ 시간/상태 변화는 의도적 에셋 전환으로

레이어 2: 샷 간 시각 연속성 (첫 프레임 설계로 담당)
  ├─ 카메라 위치 연속성 (다음 샷이 이전 샷에서 자연스럽게 이어지게)
  ├─ 조명 연속성 (같은 씬 내 조명 조건 유지)
  └─ 행동 연속성 (이전 샷 끝과 다음 샷 시작이 매치)

두 레이어 모두 Qwen3 Image의 첫 프레임 생성 단계에서 해결됨
```

### 7.5 샷 시퀀스 데이터 구조

```yaml
shot_id: 5
duration: 5s

# S 정보
S:
  scene_id: 2
  scene_purpose: "편지 모티프 첫 등장"
  emotion_beat: { start: "호기심", end: "예감" }
  character_action: "카이의 손이 편지를 향해 천천히 다가감"

# C 정보
C:
  hook_type: "unexplained_detail"
  causal_link: { from: "shot_4", to: "shot_6" }
  motif_active: "letter"
  info_disclosure: "봉투 외관만, 내용 미공개"

# V 정보
V:
  camera: { type: "ECU", angle: "slight_high", movement: "static" }
  lighting: { key: "amber_focused", fill: "minimal", ratio: "8:1" }
  composition: "rule_of_thirds, 편지 좌하단"
  mood: "tense, mysterious"

# 에셋 참조
assets:
  characters:
    - id: "kai"
      asset_version: "v1"  # 깨끗한 트렌치코트, 평온한 표정
      visible_parts: ["hands"]  # 이 샷은 손만 보임
  locations:
    - id: "cafe"
      asset_version: "a"
  props:
    - id: "letter_envelope"
      asset_version: "v1"  # 첫 등장
      first_appearance: true

# 첫 프레임 생성 (Qwen3 Image)
first_frame_generation:
  base_assets: ["kai_v1_hands", "cafe_a", "letter_v1"]
  composition_prompt: |
    "Extreme close-up of weathered hands reaching for white envelope
     on marble cafe table. Amber pendant light focused on envelope.
     Surrounding teal shadow. Slight high angle. Film noir aesthetic."
  output: "shot_5_first_frame.png"

# 영상 생성 (Hunyuan Video 1.5)
video_generation:
  input_image: "shot_5_first_frame.png"
  motion_prompt: |
    "Hand reaches slowly toward the envelope. Subtle hesitation.
     Camera remains static. Light flickers slightly."
  output: "shot_5.mp4"

# 연속성 메타
continuity:
  carry_forward_from: "shot_4"
  consistent_elements: ["lighting", "table_surface", "rain_on_window"]
  changes: ["frame_size (WS→ECU)", "focus_plane"]
```

### 7.6 연속성 체인 흐름

```
씬 시작 (씬 첫 샷):
  필요 에셋들 → Qwen3 Image로 첫 프레임 생성 (구도 새로 디자인)
  → Hunyuan Video로 5초 영상

씬 내 연속 샷:
  옵션 A (프레임 매칭이 중요한 경우):
    이전 샷 마지막 프레임 → Qwen3 Image Edit으로 카메라 이동/표정 변화
    → 다음 샷 첫 프레임 → Hunyuan Video

  옵션 B (장면 전환적 컷):
    이전 샷의 에셋들 → Qwen3 Image로 새 구도의 첫 프레임 생성
    → Hunyuan Video

씬 전환 (새 로케이션):
  새 로케이션 에셋 + 캐릭터 에셋(현재 상태) → Qwen3 Image로 첫 프레임
  → Hunyuan Video
```

### 7.7 에셋 라이프사이클

```
프로젝트 시작 시:
  S2에서 정의된 모든 캐릭터 → Qwen3 Image로 마스터 에셋 (v1) 생성
  L2에서 정의된 모든 로케이션 → Qwen3 Image로 마스터 에셋 (a) 생성

S→V 변환 시 (C적용②):
  S2 캐릭터 아크 분석 → 필요한 에셋 변형 식별
    예: 카이가 4단계 감정 변화 → asset_v1, v2, v3, v4 필요
  Qwen3 Image Edit으로 변형 에셋 일괄 생성

샷 생성 시:
  각 샷이 어떤 에셋을 쓸지 결정 (이미 라이브러리에 있음)
  → Qwen3 Image로 샷별 첫 프레임 합성
  → Hunyuan으로 영상

에셋 재생성 (사용자 요청):
  특정 에셋이 마음에 안 들면 → Qwen3로 재생성
  → 해당 에셋을 사용한 모든 샷 재생성 (자동 트리거)
```

### 7.8 샷 액션 복잡도 예산 (Action Budget)

**가장 중요한 제약**: 5초 안에 너무 많은 액션을 요청하면 영상이 무너짐.

```
샷 1개당 액션 예산:

┌─────────────────────────────────────────────────────────┐
│ 주요 액션:    1개 (필수)                                  │
│   - 5초의 핵심 사건 (캐릭터의 한 동작 or 카메라 한 움직임)  │
│   - 명확한 시작-진행-끝이 5초에 들어맞아야                  │
│                                                         │
│ 보조 액션:    0~1개 (선택)                                │
│   - 주요 액션과 충돌하지 않는 미세 동작                    │
│   - 예: "걸으면서 + 옷깃 세움" / "앉으면서 + 표정 변화"     │
│                                                         │
│ 환경 변화:    0~1개 (선택)                                │
│   - 비, 바람, 깜빡이는 빛 등 배경 무브먼트                  │
│   - 캐릭터 액션과 분리되어 작동 가능                       │
│                                                         │
│ 카메라 무브:  0~1개 (선택)                                 │
│   - 캐릭터 액션이 클 때는 카메라 정적 권장                  │
│   - 캐릭터 정적일 때 카메라 무브 가능                      │
│                                                         │
│ ✗ 위험 조합:                                              │
│   - 주요 액션 + 카메라 무브 + 환경 변화 (3개 동시) = 혼돈    │
│   - 캐릭터 2명이 각자 다른 액션 = 한쪽 무너짐               │
│   - "달려가면서 폭발이 일어나고 카메라가 뒤따라간다" = 폭망   │
└─────────────────────────────────────────────────────────┘
```

### 7.9 액션 분할 규칙 (Auto-Split)

S3에서 지정한 씬 액션이 5초 한계를 초과하면 자동 분할:

```
원본 S3 액션:
  "카이가 일어나서 코트를 입고 문을 열고 나간다"
  → 4개 액션. 5초로는 불가능.

자동 분할:
  shot_a (5초): "카이가 일어선다"           [주: 일어서기]
  shot_b (5초): "카이가 코트를 입는다"       [주: 입기]
  shot_c (5초): "카이가 문을 열고 나간다"    [주: 문 열고 + 보조: 나가기]

검증 통과:
  각 샷이 1 주요 + 0~1 보조 액션 안에 들어감
```

**분할 의사결정 트리**:

```
S3 액션 받음
  ↓
액션 개수 카운트
  ↓
  ├─ 1개 → 그대로 1샷
  ├─ 2개, 같은 캐릭터, 자연스럽게 이어짐 → 1샷 (주+보조)
  ├─ 2개, 시각적 단절 필요 → 2샷
  ├─ 3개+ → 자동 분할 (액션당 1샷 원칙)
  └─ 카메라 무브 + 큰 액션 → 카메라/액션 중 하나 우선, 다른 건 다음 샷
```

### 7.10 샷 스코프 검증 (Scope Validation)

C 적용② 단계에서 각 샷이 생성 가능한지 사전 검증:

```yaml
검증_규칙:
  - 주요_액션_수: <= 1
  - 보조_액션_수: <= 1
  - 카메라_무브_복잡도: simple (pan/tilt/dolly 1방향) | none
  - 환경_요소_변화: <= 1 (비/바람/조명 중 하나만)
  - 캐릭터_수_in_frame: 권장 1, 최대 3 (3명 이상 = 한 명 무너짐)
  - 액션_시간_추정: 3~5초 (너무 짧으면 갑작스러움, 너무 길면 잘림)

위반_시:
  CRITICAL: 자동 분할 강제 (사용자에게 알림)
  WARNING:  사용자에게 "이 샷은 무너질 위험이 있음" 경고
```

### 7.11 Start Frame 의존도 완화 전략

API 트렌드(start frame 미지원 추세)에 대비:

```
현재 (Hunyuan 1.5):
  첫 프레임 + 모션 프롬프트 → 영상

전략:
  1. 모션 프롬프트만으로도 충분히 명확하게 작성
     ("느리게 일어선다" + 첫 프레임)
     → 첫 프레임 없이도 같은 결과를 낼 수 있을 정도로

  2. 첫 프레임은 "스타일 가이드"로 격하
     영상의 시작이 정확히 첫 프레임일 필요 없음
     → 모션 프롬프트가 진짜 영상 명세

  3. 모델 교체 대비
     T2V만 지원하는 모델로 교체될 때 → 모션 프롬프트 그대로 사용 가능
     첫 프레임은 옵셔널 보강
```

## 8. 프롬프트 전략

### 정보 우선순위

프롬프트에 포함할 정보의 우선순위. 스윗 스팟까지 상위부터 채움.

| 순위 | 정보 유형 | 출처 | 이유 |
|------|---------|------|------|
| **1** | 핵심 행동 | S3 | 이것 없으면 뭘 보여줄지 모름 |
| **2** | 캐릭터 외형 | S2→L2 | 시각 일관성의 핵심 |
| **3** | 환경/로케이션 | S3→L2 | 공간 없으면 배경 랜덤 |
| **4** | 분위기/감정 | S3+C5 | 톤을 잡아줌 |
| **5** | 카메라 | C→L3 | 구도와 앵글 |
| **6** | 조명 | C→L3 | 분위기 강화 |
| **7** | 스타일 키워드 | L1 | 전체 룩 |
| **8** | 연속성 지시 | 연속성 체인 | 이전 샷과의 일관성 |
| **9+** | 추가 디테일 | L2/L3 | 소품, 날씨, 배경 인물 등 (과잉 시 충돌 위험) |

```
스윗 스팟까지:
  순위 1~4 = 거의 항상 포함 (핵심 정보)
  순위 5~7 = 포함하면 좋지만 행동과 충돌 시 제외
  순위 8   = I2V일 때 필수, T2V일 때 불필요
  순위 9+  = 스윗 스팟 내 여유가 있을 때만

스윗 스팟 확정 방법: 동일 씬에 대해 정보량을 단계적으로 늘려가며
  품질 비교 실험 수행 (TODO)
```

### 액션 작성 규칙 (핵심 행동 표현)

순위 1 "핵심 행동"은 단순 글자 수가 아니라 **액션 복잡도**가 핵심.

```
✗ 나쁜 액션 작성:
  "카이가 일어나서 코트를 입고 문을 열고 거리로 나가서 비를 맞으며 걷는다"
  → 5초로 불가능. 영상이 무너짐.

✓ 좋은 액션 작성:
  "카이가 천천히 일어선다. 옷깃이 살짝 흔들린다."
  → 1 주요 액션 (일어서기) + 1 보조 (옷깃)
  → 5초에 명확하게 들어맞음
```

**액션 작성 체크리스트**:

| 항목 | 권장 | 금지 |
|------|------|------|
| 동사 개수 | 1~2개 | 3개 이상 |
| 시간 표현 | "천천히", "갑자기" 같은 속도 명시 | "그리고", "그 다음에" 같은 순차 |
| 범위 | 한 동작의 시작-진행-끝 | 여러 동작의 연속 |
| 카메라 + 액션 | 둘 중 하나가 정적 | 둘 다 격렬한 무브 |
| 캐릭터 수 | 1명 우선 | 3명+ 동시 액션 |

### 모션 프롬프트 (Hunyuan I2V 전용)

첫 프레임은 Qwen3로 만들고, Hunyuan에는 별도의 **모션 프롬프트** 전달:

```
첫 프레임 프롬프트 (Qwen3 Image용):
  정적 묘사: 캐릭터, 환경, 구도, 조명, 분위기
  → 이미지 생성용

모션 프롬프트 (Hunyuan용):
  동적 묘사: 5초 동안 무엇이 어떻게 움직이는가
  → 영상 생성용
  → 액션 예산 규칙 적용 (1주요+1보조 이내)

예시:
  첫 프레임 프롬프트:
    "느와르 카페, 카이가 의자에 앉아 정면을 본다.
     앰버 펜던트 조명, 틸 그림자, 비 오는 창문."

  모션 프롬프트:
    "카이가 천천히 일어선다. 옷깃이 흔들린다.
     카메라는 정적. 비가 창문에 계속 흐른다."
```

## 9. 현재 파이프라인 대비 매핑

```
현재:
Story → [Pumpup] → [L1 Scene] → [L2 Shot] → [L3 Prompt] → [Video]

제안:
[Preset]→[S0~S3+Hint]→[C적용①]→[Mid Preview]→[L0~L3+BackAdj]→[C적용②]→[샷시퀀스]→[에셋]→[영상]→[조립]
                          │                                      │           │       │      │
                       S 내부 검증                            S→V 변환    Qwen3   Hunyuan
                       (인과,CDQ,핍진성)                     프롬프트생성  (이미지)  (I2V)

대응:
  현재 Pumpup        → S0~S1 + Forward Hint
  현재 L1 Scene      → S2~S3 + C적용① + Mid Preview + L0~L1
  현재 L2 Shot       → L2 + C적용②
  현재 L3 Prompt     → 샷 시퀀스 생성 (정보 우선순위 기반 프롬프트)
  현재 Video API     → 에셋 시스템 + Qwen3 첫 프레임 + Hunyuan I2V + 조립
  (신규) C적용①       → S 완료 후 서사 검증
  (신규) C적용②       → V 완료 후 S→V 변환 + 프롬프트 생성
  (신규) 에셋 시스템    → Qwen3 Image / Image Edit 기반 캐릭터/로케이션 영속성
  (신규) 두 단계 생성   → Qwen3 (첫 프레임) → Hunyuan (I2V)
  (신규) 깊이 레벨     → S0 러닝타임에 따라 전체 파이프라인 복잡도 자동 조절
```

## 10. 단계별 입출력 요약

| 단계 | 입력 | AI/모델 역할 | 출력 | 사용자 액션 |
|------|------|------------|------|-----------|
| Preset | 장르 키워드 or 레퍼런스 | S0+L0+L1 프리셋 생성 | 초기 세팅 | 수락/수정 |
| S0 | 장르, 톤, 감정, **러닝타임** | Forward Hint + **깊이 레벨 결정** | S0 확정 + D레벨 | 결정 |
| S1 | 구조, POV, 주제 | Forward Hint (D레벨에 맞춰) | S1 확정 | 결정 |
| S2 | 캐릭터, 아크, 관계 | 쉐이프/컬러 제안 | S2 확정 | 결정 |
| S3 | 씬, 감정비트, 대사 | 컬러스크립트 초안 | S3 확정 | 결정 |
| **C 적용①** | S0~S3 전체 | **인과/CDQ/핍진성/클리셰 검증** | S 검증 리포트 | 수정/수락 |
| Mid Preview | S+C검증 결과 | V축 전체 추천 | V 방향 합의 | 수락/수정/S수정 |
| L0+L1 | Mid Preview 기반 | 기술 파라미터 | L0+L1 잠금 | 확인 |
| L2 | S→V 자동 매핑 | 디자인 시스템 생성 | L2 확정 | 수정 가능 |
| L3 | S→V 자동 매핑 | 샷별 파라미터 | L3 확정 | 수정 가능 |
| **C 적용②** | S+V 전체 | **S→V 변환, 프롬프트 생성, 액션 스코프 검증, 자동 분할** | 샷 시퀀스 (프롬프트 + 분할된 샷) | 검토/수정 |
| **에셋 마스터** | S2 캐릭터 + L2 로케이션 | **Qwen3 Image** (text-to-image) | 마스터 에셋 라이브러리 (v1, a) | 검토/재생성 |
| **에셋 변형** | 마스터 + 캐릭터 아크/씬 변화 | **Qwen3 Image Edit** | 변형 에셋들 (v2, v3, ...) | 검토/재생성 |
| **첫 프레임** | 샷 메타 + 필요 에셋들 | **Qwen3 Image** (composition) | 샷별 첫 프레임 이미지 | 검토/재생성 |
| **영상 생성** | 첫 프레임 + 모션 프롬프트 | **Hunyuan Video 1.5 (I2V)** | 5초 클립 | 재생성 (에셋 or 프롬프트 수정) |
| **조립** | 클립 시퀀스 | 조립 + 트랜지션 (사운드 후일) | 최종 영상 (무음) | 최종 확인 |

> 사운드/음악 단계는 별도 문서 `sound_ideation.md` 참조 (현재 미정)
