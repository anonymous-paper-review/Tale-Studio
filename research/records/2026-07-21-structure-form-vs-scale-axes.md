# Writer / S1 깊이 사다리 — 형태×규모 축 분석

> 2026-07-21 후속 토론 기록. 선행: `2026-07-21-narrative-structure-session.md` (같은 날 세션의 실측을 재해석).
> **성격 주의**: 이 문서는 기존 실측의 재해석 + 설계 분석이다. §7의 프로브를 아직 안 돌렸으므로 "지뢰가 실제로 터지는가"는 미검증.

---

## 0. 한 줄 요약

S1 시스템프롬프트의 두 리스트(구조 유형 가이드 / 깊이 레벨 사다리)는 **서로 다른 축**인데 어휘가 섞여 있다:

- **구조 유형 = 형태(form)**: 긴장·의미가 조직되는 패턴. **스케일-프리** — 15초 광고가 완전한 기승전결일 수 있고, 2시간 영화가 단순 3막일 수 있다.
- **깊이 레벨 = 규모(scale)**: 비트 수·서브플롯 수·층위. **길이와 물리적으로 결합된 건 이쪽뿐** (서브플롯 하나 = 설치–전개–회수할 스크린타임 필요).

문제: 사다리가 규모를 말하는 자리인데 **형태 어휘를 빌려 씀** — D1 "구조 없음", D2 "setup → action → result"(그 자체가 미니 선형 3박자 → 기승전결 콘텐츠와 정면충돌), D3 "(3-act 또는 기승전결)". 그래서 '3-act'가 두 리스트에 이중 소속, '구조'가 두 뜻(형태/규모)으로 사용됨.

단, 기존 실측을 다시 읽으면 이건 **이미 터진 버그가 아니라 아직 안 밟은 지뢰**다 (§2).

---

## 1. 발단

선행 세션 문서를 읽고 나온 질문(xcape):

- 영상이 짧아도 기승전결이 담길 수 있고, 길어도 3막일 수 있고 루프물일 수도 있다.
- 그런데 프롬프트는 구조유형 가이드(서사 특성 함축 리스트)와 깊이 레벨(사실상 길이 매핑)에 '3-act'와 '구조'를 양쪽에 다 써놨다 → 모델이 헷갈렸나?

참고(코드 사실): `Genre`에는 `runtime_seconds`와 `depth_level`이 **별도 필드**로 존재 (`src/lib/writer/types/pipeline.ts` Genre). depth가 코드에서 길이로부터 계산되는 건 아니고, 실험 프리셋이 관행적으로 짝지음(shorts 15s·D1, feature 6000s·D7).

---

## 2. 기존 실측의 재해석 — "헷갈렸나?"는 반만 맞다

선행 세션 데이터(§2b·§2c)를 이 프레임으로 다시 읽으면, 모델은 structure_type 선택에서 사다리를 **거의 무시**했다:

| 실측 (선행 세션) | 사다리 지시 | 실제 출력 |
|---|---|---|
| shorts D1 | "구조 없음" | 3-act 3막 (무시) |
| feature D7 | 다층+서브플롯 다수 (non-linear 언급 없음) | non-linear 3막 (무시) |
| kishoten D3 | "(3-act 또는 기승전결)" | 기승전결 4막 |

그리고 관측된 유일한 오분류(loop→3-act)는 **사다리를 안 건드리고** 앵커 삭제 + circular 설명 보강만으로 해결됨 — 격리실험이 "주범은 앵커"임을 입증. 즉 사다리 어휘 충돌은 지금까지의 편향 원인이 **아니었다**. 결함은 실재하되(§3), 현재는 잠복 상태.

---

## 3. 사다리의 결함 3개 (코드 근거 있음)

1. **상호모순 지시의 동거**: `s1_structure.ts`의 능동 판별 지시("억지로 3-act에 끼워 맞추지 마라")와 D2·D3 줄이 반대 방향으로 당김. 현재는 전자가 이기는 균형일 뿐 — **모델 bakeoff로 프로바이더 교체 중이라 균형이 뒤집힐 수 있는 취약 구조**.
2. **D1 약속 vs 출력 스키마 모순**: "구조 없음, CDQ 생략 가능"이라 해놓고 출력은 `structure_type`·`acts`·`central_dramatic_question` 전부 요구. 모델이 D1 지시를 무시해서 우연히 무사한 것이지 설계 정합이 아님.
3. **사다리 절반이 dead letter**: D4~D7의 실제 페이로드는 서브플롯 수인데 `NarrativeStructure` 출력엔 서브플롯 필드가 없음 → 실릴 곳이 없다. 선행 세션 §2b "depth 실효 약함"과 정확히 합치. 서브플롯/규모 레버는 애초에 scenes(`sceneCountHint`) 관할일 가능성.

---

## 4. 과교정 금지 — 두 축은 완전 직교가 아니다

결합 제약은 실재한다. 수정 시 커플링 전부 삭제가 아니라 **명시적 제약으로 분리**해야 함:

- hero's journey: 최소 비트 수 필요 → 초단편에선 사실상 불가/압축 변형.
- non-linear: 재배열할 단위(씬) 3개 이상 있어야 성립 → D1에선 무의미.
- 서브플롯: 개당 스크린타임 소요 → 러닝타임이 상한을 만든다 (단, 하한은 아님 — 긴 영상이 단순해도 됨).

**⚠ 선행 세션 열린 질문 #2에 대한 경고**: "depth→act 복잡도 매핑을 프롬프트에 추가"는 이 프레임에선 같은 축 혼합을 반대 방향으로 반복하는 것. **막 수는 형태의 소유물**(3-act=3, 기승전결=4). depth가 키울 것은 막 수가 아니라 막 안의 비트 밀도·서브플롯·층위.

## 5. 층위 한 겹 더 — 전제(story) vs 담론(discourse)

"루프물은 장르적 특성 아닌가"라는 직감이 맞다:

- 타임루프 = **전제(premise, 스토리 내용)**. 수미상관 = **담론 구조(discourse)**.
- 사랑의 블랙홀·Edge of Tomorrow는 타임루프물이지만 막 구조로는 표준 3-act.
- 현재(수정 후) circular 설명은 "시간 루프·데자뷔·수미상관"을 한 줄에 합쳐놓음 — 단편 프로브에선 무해했지만 **장편 타임루프 입력에선 오분류 소지**.
- 더 깊은 냄새: 유형 리스트 자체가 비-분할(non-partition) — 막형태(3-act/기승전결) / 성장 원형(hero's journey) / 시간 배열(non-linear) / 장치(circular)가 섞여 있고 상호배타가 아님(예: hero's journey는 대개 3-act이기도; non-linear는 다른 형태 위에 오버레이 가능). 단 소비자 5곳이라 스키마 수술은 비쌈 → **지금은 인지만**.

---

## 6. 수정 방향 (제안, 미실행)

1. 사다리에서 형태 어휘 제거 — 규모 언어만: 비트 수·서브플롯 수·층위. (판별 테스트: "이 줄에서 형태 단어를 다 지워도 문장이 성립하나")
2. 형태×규모 결합 제약을 별도 2~3줄로 명시 (§4 목록).
3. D1의 "구조 없음/CDQ 생략"은 스키마와 합의 필요 — 출력 어휘에 없는 약속을 프롬프트에 쓰지 않기 (acts 1개 허용 등으로 규모 언어화하거나, 스키마에 반영하거나).
4. D4~D7의 서브플롯 지시는 S1 잔류 vs scenes 이관 판단 — S1 출력에 실을 필드가 없다면 이관이 자연스러움.
