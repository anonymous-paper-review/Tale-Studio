## 0. 요약

**전역 발견 5개** (개별 스테이지보다 크다):

1. **물리 상수 산포** — 같은 백엔드 법칙이 스테이지마다 다른 숫자로: 샷 초 decoupage `2~8s(예외 10)` vs V4 헤더 `5~15초` vs V4 본문 `2~8초` vs S3 `5초 한 샷에 한 액션`; 모션 프롬프트 V4 `50~80자` vs V5 fallback `50~100자`. `[코드]` → P2 위반. 한 곳(코드 상수)에서 주입해야 한다.
2. **시간 예산의 책임 스테이지 부재** — runtime 정합을 아무도 소유하지 않는다: S3는 "총합 ≈ N초" 산문 한 줄, decoupage validator는 샷당 2~8s만, C2는 total_duration_seconds를 계산만 하고 **runtime과 비교하는 코드가 없다** `[코드]`. 장편에서 조용히 붕괴하는 구조 (E3).
3. **죽은 배선 위의 프롬프트** — storyCheck·midPreview는 skip 설정 경로가 없어 **프로덕션에서 항상 skip** `[코드: resolveSkip 기본 true + skip: 설정하는 호출자 grep 0건]`. 그런데 V0 프롬프트는 "Mid Preview 제안을 채택"이라 말하고(항상 빈 seed), Compact Mode는 비활성(`COMPACT_DEPTH_LEVELS=[]`)인데 V4에 Compact 분기 서사가 남아 있다. P4 위반군.
4. **만들고 버리는 출력** — C2의 hook_type(10종 분류)·motif_active: 소비처 전무 `[코드: grep — types 외 0건]`. midPreview의 color_script/emotional_arc/production_difficulty: 소비처 전무 `[코드]`. V4의 story_beat_ref: 코드가 decoupage 값으로 덮어씀 `[코드: v4_shots.ts 샷 병합부]`.
5. **클리셰 정책 자기모순** — 제품 본질은 "모호하면 관습(클리셰) 디폴트"인데 C1은 클리셰를 세고 감점한다. 어느 프롬프트에도 "빈칸은 관습대로 채워라"는 지시가 없다 `[코드]`. P1/P5 위반 — 본질 문장이 프롬프트 아키텍처에 부재.

**감사 판정 한 줄**: 파이프라인의 코드 레이어(검증·정합·복원)는 성숙한데, 프롬프트 레이어는 여러 차례의 재설계(producer 게이트, V축 재설계, decoupage 도입, skip 기본화)를 **따라오지 못한 퇴적층**이다. 재작성의 최우선은 새 내용 추가가 아니라 **퇴적층 제거 + 원장 3블록 표준화**.

## 1. 스테이지별 감사

표기: 유형·파급은 `docs/writer-llm-model-strategy-2026-07-21.md` §2의 분류 재사용.

### 1.1 S1 narrativeStructure (🔷저작·🌳뿌리)

선행 문서 2건이 이미 심층 분석 — 여기선 포인터만: 앵커 버그 수정 완료(`../narrative/2026-07-21-narrative-structure-session.md` §3), 깊이 사다리의 형태×규모 축 혼선 + 결함 3개 + 프로브 2셀(`../narrative/2026-07-21-structure-form-vs-scale-axes.md`).

추가 발견 없음. 재작성 방향은 축 분석 문서 §6 그대로.

### 1.2 S3 scenes (🔷저작·🌿가지)

**잘된 것**: 오픈 캐스트/로케이션 규칙(원장 원칙의 모범) + 코드 정규화(`normalizeSceneLocations`) + act 커버리지 repair 루프 `[코드]`. 로케이션 언어 규칙(스토리 언어로 작명)도 원장 정신.

**문제**:
- **① 시간 산수 위임 (전역-2의 진원지)**: 씬 수 힌트는 depth에서(`sceneCountHintMap`), 총초는 runtime에서 — 둘의 정합을 모델이 암산해야 한다. feature(6000s·D7)면 30+씬 × 200s/씬인데, 동시에 "한 씬은 보통 1~3 액션, 5초 한 샷에 한 액션"을 요구 → **한 씬의 액션 콘텐츠는 5~15s 분량인데 씬 길이는 200s여야 하는 모순** `[코드에서 유도한 가설 — E3]`. 모델은 estimated_seconds를 뻥튀기하거나 총합 미달로 타협할 수밖에 없다.
- **② 죽은 명칭**: "주어진 S0/S1/S2 위에서" — s0/s2는 producer 게이트로 대체됨(steps.ts §4 주석). 모델에게 존재하지 않는 상류를 암시. P4. `[코드]`
- **③ 씬 purpose 10종에 분포 지침 없음**: 전 씬이 conflict여도 규칙 위반이 아니다. 관습 디폴트 원칙이라면 "이 장르의 전통적 씬 흐름"(예: 광고=hook→product→CTA, 드라마=setup→complication→…)이 더 유효한 가이드. `[가설 — E4]`

**피드백·의도**: (①) 예산표를 코드가 계산해 주입 — `씬 수 N±α, 씬당 s_min~s_max초, 총합 T±10%` 형태. 의도: P6 — 모델의 강점은 배분 판단이지 제약 충족 암산이 아니다. (②) 명칭 현대화. (③) 장르별 관습 씬 흐름 힌트를 옵션으로.

**실험 E3 (시간 예산)**: 하네스 `feature` 프리셋으로 scenes 실행(세션 열린질문 #1과 동일 run에서 관찰 추가) → (a) total_estimated_seconds vs 6000 편차율, (b) **씬 내부 정합**: estimated_seconds vs scene_actions 수 × 5~8s의 괴리 분포. 예산표 주입판과 A/B. 판정: 주입판이 (a) 편차 ±10% 이내로 수렴 + (b) 괴리 중앙값 감소하면 채택.

**실험 E4 (purpose 분포)**: DB 감사(`writer_runs.state->scenes`)로 purpose 히스토그램 — revelation/payoff/setup이 실제로 나오는지, conflict/exposition 편중인지. 편중이면 ③의 관습 흐름 힌트 실험 가치 상승.

### 1.3 C1 storyCheck (🔍검수 — **프로덕션 항상 skip**)

**문제**:
- **① 죽은 스테이지**: `resolveSkip` 기본 true + `skip:`을 설정하는 호출자 0건 `[코드]` — 모든 실 run에서 빈 리포트. 존재 이유가 결정 안 된 상태로 코드만 생존.
- **② 클리셰 정책 모순 (전역-5)**: "클리셰 감지 → 개수 세기 → WARNING"은 제품 본질(클리셰=디폴트)과 정면 충돌. 잡아야 할 것은 진부함이 아니라 (a) 인과·핍진성 붕괴 (b) 유저 차별점의 실종.
- **③ 장식 파라미터**: `retryCount` — 재시도 루프 호출자 없음 `[코드]`.
- **④ 룰로 가능한 것의 LLM 위임**: CDQ 존재 여부·yes/no 형식은 룰 검사 가능. LLM은 "속성 5개 충족" 같은 의미 판단만.

**피드백·의도**: 살리려면 "차별점 보존 검사기"로 재정의 — 유저 story의 고유 요소(비클리셰 성분)를 추출해 scenes에 생존했는지 검사. 죽이려면 코드도 정리. 중간 상태(코드 생존+항상 skip)가 최악 — 프롬프트 감사 비용만 발생. 의도: P4/P5.

**실험 E5**: 같은 입력 10 run × C1 on/off — C1이 CRITICAL을 낸 run과 최종 샷 시퀀스 품질(사람 rubric: 인과 성립·차별점 생존) 상관 측정. 상관 없으면 제거 확정, 있으면 default on + ② 재정의.

### 1.4 midPreview / bridge (🔷저작 — **프로덕션 항상 skip**)

**문제**: ① 항상 skip이라 V0~V4가 전부 빈 seed로 동작 `[코드]` — "S↔V 협상"이라는 설계 의도가 배선에서 소멸. ② color_script·emotional_arc_visualization·production_difficulty·warnings 전부 소비처 없음 `[코드: grep]` — 켜져 있어도 절반이 버려진다.

**피드백·의도**: 삭제가 자연스럽다 — V0가 이미 genre에서 직접 결정하고 있고(현실), 씬별 색은 V2/V3가 소유한다. "양방향 협상"이 정말 필요하면 그건 스테이지가 아니라 **유저 확인점**(producer 게이트처럼)이어야 한다 — LLM끼리의 협상은 협상이 아니라 한 번 더 저작일 뿐. 의도: P4 + 아키텍처 §0 (파생 가능한 값은 만들지 않는다).

**실험 E6**: midPreview on/off × 4 프리셋 → V0~V4 산출 차이를 rubric(스타일 정합·씬 색 계획 품질)으로 비교. 차이 무의미하면 삭제 확정. (예상: 무의미 `[가설]` — seed가 1~2문장 힌트에 불과)

### 1.5 V0 visualIdentity (🔷저작·🌳뿌리)

**잘된 것**: styleAnchor 처리 — "유저 확정값은 발명 대상이 아니라 입력" `[코드]`. **파이프라인 전체에서 원장 원칙이 가장 정확히 구현된 곳.** P1의 원형으로 삼을 것.

**문제**: ① genre.format(유저 확정 화면비)→ aspect_ratio 강제 규칙이 프롬프트에 없다 — genre JSON이 보이니 아마 따르겠지만 재량에 맡겨진 원장 값 `[코드]`. 애초에 이건 LLM이 정할 값이 아니라 코드가 복사할 값. ② "Mid Preview 제안을 채택하되" — 항상 빈 seed (P4).

**실험 E2**: `vertical_9:16` genre × 10 run → aspect_ratio 이탈률. **이탈 0이어도 코드 고정으로 이관** — 원장 값의 LLM 경유는 이탈률과 무관하게 설계 냄새(모델은 제안만 한다 원칙의 역방향 적용: 확정값은 모델을 거치지 않는다).

### 1.6 V1 actVisualArc (🔷저작)

**문제**:
- **① "≤10분 영상 전제" 하드코딩** `[코드]` — feature(100분)와 모순. S1 사다리와 같은 병: 길이 결합이 산문에 박힘.
- **② 시맨틱 미착지**: v1의 페이로드는 "막 진행에 따른 시각 변화"(시간축)인데, 소비처인 v2는 **막 불변(전역) 산출물**(팔레트·로케이션·의상)만 낸다. V3(씬별 lighting_arc를 정하는 곳)는 v1을 입력받지 않는다 `[코드: v3_scene_plan.ts 시그니처]`. 즉 막별 아크는 v2의 참고 컨텍스트로 소비되고 **시간축 정보는 씬/샷 레벨에 도달하지 못한다**. 모델전략 문서 §4의 "사장" 분류는 절반 정확 — 정확히는 "소비되되 착지 없음".

**피드백·의도**: 살리려면 V3가 씬의 act_ref로 해당 막 아크를 받아 lighting_arc/palette_emphasis에 상속시켜라(배선 복원). 죽이려면 v2 입력에서도 빼고 콜 제거(6.8s). 중간이 최악. 의도: 만들고 버리는 출력 금지(전역-4).

**실험 E8**: v1 유/무 A/B → V3 산출의 막별 분산(1막 씬들 vs 3막 씬들의 lighting_arc·palette 차이) 측정. 현 배선에선 차이가 안 날 것 `[가설 — 코드 리딩상 거의 확정이나 산출로 확인]`. 배선 복원판(V3에 act arc 주입)에서 차이가 나고 그게 보기 좋으면 복원, 아니면 제거.

### 1.7 V2 design (🔷저작·🌿가지)

**잘된 것**: 원천 보존 ID 정합(입력 ID를 권위 목록으로 결정론 정렬) `[코드]` — §5 원칙의 모범.

**문제**: 색 어휘 불통일 — v2 palette는 hex, color_meaning은 색이름 키, V3 palette_emphasis는 hex, (죽은) midPreview color_script는 색이름 `[코드]`. T2I 프롬프트까지 일관 주입되려면 한 가지 표기로.

**실험**: 우선순위 낮음. DB 감사로 palette 필드 표기 산포만 확인해 두기.

### 1.8 V3 sceneCinematography (🔷저작·🌿가지)

**잘된 것**: validator+repair 루프, 필름 문법 어휘(coverage/lens/energy/rhythm 가이드)의 밀도 — 관습(prior)을 프롬프트에 넣는 좋은 예.

**문제**:
- **① shot_count_target의 권위 사슬 모순**: V3가 정하고 → decoupage는 "참고용 힌트일 뿐 제약 아님"으로 강등하고 → V4는 "V3.shot_count_target ±1" 준수를 요구한다 `[코드: 세 파일]`. 데쿠파주 확정 모드에선 V4의 샷 수가 decoupage로 고정되는데도 discipline 섹션에 ±1 문구가 그대로 들어간다 — **한 프롬프트 안에 상호모순 지시 동거** (S1 사다리와 같은 genus).
- ② Compact 분기: 비활성(`COMPACT_DEPTH_LEVELS=[]` `[코드]`)인데 V4·infer_v3에 서사 잔존. 재활성 옵션은 코드 주석으로 충분 — 프롬프트에서 는 정리 (P4).

**피드백·의도 (급진안 포함)**: V3와 decoupage는 같은 감독 판단의 두 zoom이다 — 씬 디시플린(어떻게 찍나)과 샷 분해(몇 컷으로)를 다른 콜로 나눈 대가가 권위 모순 + 30.2s. **decoupage에 흡수**해 씬당 한 번에 "디시플린+샷 리스트"를 저작하고 V3 validator를 그 출력에 적용하는 안을 실험할 가치. 의도: 직무 모델 §2 — 경계 정당성 3조건(유형/모델/체크포인트) 중 어느 것도 이 경계를 요구하지 않는다.

**실험 E9 (흡수 A/B)**: 통합 프롬프트(디시플린+데쿠파주) vs 현행 2단 — 4 프리셋 × rubric(샷 수 적정·리듬 다양성·180°축/렌즈 일관·added 샷의 질). 판정: 통합판이 rubric 동등 이상 + 레이턴시 절감이면 채택. (실측 시간: V3 30.2s + decoupage 107.8s — 통합 시 서버리스 240s 예산 내 여유 `[실측: 모델전략 §2]`)

### 1.9 decoupage (🔷저작·🌳뿌리) — **파이프라인 최고의 프롬프트**

원칙이 물리에 근거하고("이유 없는 무빙 = 둥둥 떠다니는 영상", "리듬 = 뇌 아픈 영상의 해독제"), 연산(derived/added/merged/split)이 명확하고, added를 "연출의 본질"로 격상한 것 모두 정확하다. 재작성 시 이 프롬프트의 톤·구조(원칙 → 연산 → 문법 → 제약)를 다른 저작 스테이지의 템플릿으로.

**남은 문제**: ① uncovered_beats를 프롬프트만 요구하고 검증 코드 없음(coverage_ratio 기록뿐) `[코드]` — REMOVE 남용이 조용히 통과 가능. ② 샷 초 대역 2~8s의 단일 소스화(전역-1)에 이 정의를 기준으로.

**실험**: 품질 투자 1순위(Sonnet A/B)는 모델전략 문서 §4에 이미 계획됨 — 중복 설계 안 함. 추가로 E10: DB 감사로 uncovered_beats 분포(비트 커버리지 실태) 확인 후 validator 추가 여부 결정.

### 1.10 V4 shotDesign (⚙️수행·🍂잎)

**문제**:
- **① 초 대역 자기모순**: 같은 프롬프트 안에서 "V4c: 5~15초의 동적 변화"(3분할 정의부)와 "1 샷 = 2~8초"(샷 분배 원칙) `[코드: v4_shots.ts systemInstruction]`. 5~15는 구세대 잔재로 추정. 전역-1의 대표 사례.
- **② 시키고 버리는 요구**: "story_beat_ref로 scene_actions에 1:1 매핑" — 데쿠파주 4연산(merged=[i,j], added=[])과 개념 모순이고, 코드가 어차피 dec.source_beats로 결정론 덮어쓰기 `[코드]`. 모델의 주의를 낭비시키는 죽은 지시.
- ③ ±1 문구 동거(1.8-①과 동일 건).
- ④ asset_version 판정 기준 빈약("의상/감정/외형 변화" — 어느 정도가 변화인가) — 씬 간 일관성을 C2 검증에 떠넘김.
- ⑤ **언어 정책 부재**: first_frame_prompt/motion_prompt의 언어 미지정. 실측에선 영어로 나옴(모델전략 §1b) `[실측]`, V5 fallback은 "한글"을 명시 `[코드]`, persist는 EN/native를 LLM으로 파생(40~60s) `[코드: steps.ts 주석]`. 세 곳이 제각각.

**피드백·의도**: (⑤) 렌더 프롬프트는 **영어를 정본으로 저작**하고 native는 표시용 파생으로 명시하는 게 자연스럽다 — 백엔드 성능 + persist 파생 비용 절감. 단 백엔드별 한/영 성능차는 실측으로. (①②③) 퇴적층 제거. 의도: P2/P4.

**실험 E7 (대역 통일 회귀)**: 대역 문구 통일 후 4 프리셋 → duration_seconds 분포 before/after (5~15 잔재가 실제로 긴 샷을 만들고 있었는지도 이때 드러남). **실험 E11 (프롬프트 언어)**: 같은 샷 스펙으로 T2I/TI2V를 한글 vs 영어 프롬프트 생성 → fal 백엔드 산출 품질 블라인드 비교 + persist 파생 시간 측정. 판정: 영어 정본이 품질 동등 이상이면 채택(파생 1회 절감).

### 1.11 C2 shotCheck (🔍검수)

**잘된 것**: 결정론 우선 진화 — LLM 조립 skip(>24샷) + reconcile + asset ref 정규화 + split 누락 블록 상속 `[코드]`. 직무 6(계약 검증·조립)의 올바른 방향.

**문제**:
- **① 만들고 버리는 저작 (전역-4)**: Step1 프롬프트의 hook_type 10종·motif_active·continuity 상세 — 소비처 전무 `[코드: grep]` + >24샷이면 그 LLM 콜 자체가 skip + 결정론 복원분은 이 메타가 근사값. 즉 **정교한 분류 체계가 소형 프로젝트의 일부 샷에만, 아무도 안 읽는 형태로 존재**.
- ② Step2(Claude) 검증 중 동사 수·순차 표현("그리고" 검출)은 룰 선별 가능 — LLM은 "첫 프레임 부정" 같은 의미 판단만.

**피드백·의도**: "재미 훅"은 본질(관객 기대) 관점에서 좋은 개념이지만, **조립 단계에서 후처리 라벨링하는 건 저작이 아니다**. 살리려면 S3(씬 훅)나 decoupage(샷 훅) 저작 시점으로 올리고 소비처(director UI 배지? 편집 제안?)를 먼저 배선하라. 아니면 Step1 프롬프트에서 제거해 조립을 완전 결정론화(LLM 콜 1개 제거). 의도: 아키텍처 §5 — 소비자 없는 산출물은 만들지 않는다.

**실험 E12**: DB 감사 — 기존 run들의 hook_type 채움률·분포(LLM 조립이 돈 소형 run 한정). 데이터가 무의미하면 제거 쪽 근거. + C2 Step1 완전 결정론화 A/B: 최종 샷 시퀀스 필드 충실도 비교(어차피 reconcile이 진실이면 차이 없어야 함 `[가설]`).

### 1.12 V5 renderPrompts (⚙️수행·🍂잎)

**잘된 것**: 추출 우선 + LLM은 결손 보충만 `[코드]` — P6 정신.

**문제**: ① fallback 프롬프트 "한글, 200~400자"/"한글, 50~100자" — V4와 언어·자수 불일치(1.10-⑤, 전역-1). ② fallback 빈도 미계측 상태로 방치 가능 — extraction_summary가 이미 기록하니 DB 감사만 하면 됨.

**실험**: E11에 합류(언어) + extraction_summary 집계(신규 실험 불요).

## 2. 급진 재구조 제안 (원작자 의도 단순 가정 — 구조 갈아엎기 레벨)

경계 정당성 기준(독트린 §2): 스테이지 경계는 (a) 저작→수행 전환, (b) 모델 배치 전환, (c) 체크포인트 단위일 때만 정당.

- **A안 (보수 — 재작성만)**: 스테이지 유지, P1~P6 적용해 프롬프트 전면 재작성(퇴적층 제거 + 원장 3블록 + 상수 주입). 스키마·배선 불변. **어떤 안을 가더라도 A는 선행 조건.**
- **B안 (중도 — 경계 3개 제거)**:
  1. S1+S3 병합 (세션 열린질문 #5와 동일; 같은 저작·같은 모델·항상 연속 실행 + act_ref 정합 비용 유발 — 경계 정당성 0/3. 실측 9.8s+20.1s=30s로 예산 무관)
  2. V0+V1+V2 병합 → "아트 바이블" 1콜 (persist 시점이 v2 뒤라 지연 손실 없음 `[코드]`; 4.1+6.8+29.7s=41s)
  3. V3→decoupage 흡수 (1.8 참조)
  - 결과: LLM 저작 콜 12→7. 각 병합은 독립 실험 가능(E13·E9).
- **C안 (급진 — 직무 그대로 4계층)**: 저작 3콜(서사 바이블 / 아트 바이블 / 씬별 감독) + 수행 1콜(V4 spec 부착) + 코드 조립(C2/V5 결정론화). 검수는 룰 우선 + 의미 스팟체크 1콜 이내.
  - 리스크: 한 콜에 결정이 몰리면 주의 분산으로 개별 결정 품질 저하 `[가설]` — 병합은 반드시 품질 rubric A/B 후 채택. 체크포인트 굵어짐(서버리스 재시도 단위 커짐)은 실측 시간상 여유 있으나 장편에서 재검증 필요.

**의도**: 콜 경계마다 정보 손실(JSON 직렬화로 뉘앙스 소실)·모순 기회(권위 사슬)·레이턴시가 생긴다. 현 12콜 중 경계 정당성을 가진 것은 decoupage→shotDesign(저작→수행, 모델전략 문서도 유일한 진짜 경계로 지목)과 검수 진입뿐이다.

**실험 E13 (S1+S3 병합)**: 병합 프롬프트(구조+씬 한 번에, 출력에 {structure, scenes} 유지 — 소비자 5곳 스키마 보존) vs 현행 2단 — 4 프리셋 × (act 커버리지 위반율, 씬 품질 rubric, 총 레이턴시). 세션 열린질문 #5의 구체화.
