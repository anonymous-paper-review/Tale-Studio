# 판정 대기 실험 5건 — 한눈 비교표 (일괄 판정용, 2026-07-21)

> **용도: 제품 오너가 5건을 한 번에 판정하는 문서.** 5건 전부 측정은 끝났고 제품 동작은 아직 아무것도
> 바꾸지 않았다 — 판정 전까지 실서비스는 이 실험들의 영향을 전혀 받지 않는다.
> 각 실험의 입력·출력 원문과 자세한 근거는 개별 결과 문서(각 행에 링크)가 진실이다.

## 1. 한눈 비교표

|---|---|---|---|---|---|
| **E6** [상세](E6-midpreview.md) | 항상 꺼져 있는 "밑그림 미리 그리기" 단계, 켜면 나아지나? | 켠 쪽이 **오히려 불안정** — 같은 입력인데 실행마다 매체(3D↔실사)가 바뀌고, 표준 단어로 써야 할 칸이 긴 자유 문장으로 풀어짐. 실행마다 9~10초 손해 | **삭제** | 밑그림 단계 코드와 그걸 참조하던 문구들 정리 | 없음 — 항상 꺼져 있던 현실이 이미 정답이었다 |
| **E8** [상세](E8-v1-arc-wiring.md) | 매번 만들고 버려지는 "막별 색·조명 설계", 실제로 쓰이게 이어주면 효과 있나? | **있다** — 막 사이 조명 온도 진행폭 약 2배, 막마다 쓰는 색 종류 3색→6~11색 분화. 검증 위반은 안 늘었다 | **배선 복원** | 연결 코드 1줄 (실험에 쓴 코드가 그대로 구현체) | 이번엔 임시 고정 색 목록으로 실험해서, 실제 전체 색 계획 안에서의 변주인지 1회 확인 필요. 제거를 고르면 실행당 ~7초 아끼는 대신 막별 화면 톤 진행 능력 포기 |
| **E9** [상세](E9-v3-absorb.md) | "씬 촬영 계획"과 "샷 나누기"를 따로 두 번 시키는 걸 한 번으로 합쳐도 되나? | 시간 **~40% 절감** + 호출 1회↓, 샷 길이 규율 동등. 대가: **샷이 24% 더 만들어짐**(샷 수 = 하류 이미지·클립 생성 비용). 부수 발견: 지금도 샷 나누기는 촬영 계획이 정한 샷 수 목표를 **안 지키고 있었다**(매번 3~4개 초과) | **조건부 통합** | 이미 코드에 있는 "한 번에 시키기" 경로를 전 구간으로 확장 | ① 씬마다 샷 개수 상한 힌트를 같이 줘서 샷 불어남 통제 (기존 메커니즘 확장 — 신규 개발 아님) ② 촬영 계획 산출물을 참조하던 다른 단계들의 정보 출처를 옮기는 설계. 유지를 골라도 "목표를 안 지키는 모순"은 별도 수선 필요 |

## 2. 판정끼리의 상호작용 (같이 결정하면 좋은 것)

- **E9와 E13은 같은 방향의 구조 결정**이다 (LLM 호출 12회→7회로 줄이는 로드맵의 몸통). 둘 다 채택하면
  파이프라인이 크게 줄지만, "정식 구현 후 재검증"이라는 같은 종류의 후속 작업이 두 건 생긴다. 하나만
  먼저 간다면 리스크가 작은 쪽은 E9 — 합쳐진 경로가 이미 제품 코드에 존재한다(짧은 영상용으로).
- **E8 채택과 E6 삭제는 충돌하지 않는다** — 서로 다른 연결선이고 실험도 분리 측정했다.
- **E11의 "언어 명시"는 판정과 무관하게 필요하다** — 한국어 정본을 골라도 명시 지시는 넣어야 한다
  (지금은 모델이 언어를 임의 선택). 이것만은 실험 없이 적용 가능한 청소성 수정으로 분리할 수 있다.

## 3. 판정 후 실행 순서 제안 (참고용)

1. **즉시형** (판정만 하면 반나절 내): E6 삭제 / E8 연결 1줄 / E11 언어 명시
2. **재검증 동반형**: E9 (샷 개수 힌트 확장 + 참조 이관 설계) → E13 (정식 구현 + 뒷정리 이관 + 재검증)
3. E8 채택 시 선행 확인 1회: 실제 색 계획 단계와 연결한 상태에서 "전체 색 목록 안 변주" 준수율

## 4. 이 문서로 판정하는 법

각 행에 대해 3택: **채택 / 미채택 / 조건 변경 후 재측정**. 판정 결과는 이 파일이 아니라
각 결과 문서 + 계획 문서 상태표에 기록한다 (진실은 개별 결과 문서 규칙 유지).
