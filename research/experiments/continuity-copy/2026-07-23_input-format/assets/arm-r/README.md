# R 상한 대조군 — 이 팔의 재료는 어디에 있나

> 정의: [`../../conti.md`](../../conti.md) §3 · 방식이 아니라 **비교용 눈금**이다.
> **요체: 원본 영상의 실제 프레임을 시작·끝 프레임으로 그대로 영상 모델에 넣는다.** 우리 이미지
> 생성 계층(정본→시트→프레임)을 완전히 우회하므로, R의 결과 = "영상 모델이 낼 수 있는 최고치".
> A~C가 R에 얼마나 근접하는지로 **이미지 계층에서 잃은 것**과 **영상 계층에서 잃은 것**을 분리해 읽는다.

## 이 폴더에 이미지 파일이 없는 이유

입력이 전부 **원본에서 이미 추출해 둔 콘티 프레임**(`../conti/`)이라 새로 만들 것이 0장이다
(생성 0콜). payloads.json이 그 파일들을 직접 가리킨다:

| 샷 | start_image | end_image | 길이 |
|---|---|---|---|
| 1 | `conti/s1_start.jpg` | `conti/s1_end.jpg` | 5.53s |
| 2 | `conti/s2_start.jpg` | `conti/s2_end.jpg` | 3.43s |
| 3 | `conti/s3_start.jpg` | `conti/s3_end.jpg` | 1.44s |
| 4 | `conti/s4_start.jpg` | `conti/s4_end.jpg` | 3.30s |
| 5 | `conti/s5_start.jpg` | `conti/s5_end.jpg` | 3.63s |
| 6 | `conti/s6_start.jpg` | `conti/s6_end.jpg` | 1.57s |

(`conti/sN_mid.jpg`는 입력이 아니다 — 원본 동작 궤적을 눈으로 대조하는 리뷰용 참고 이미지.)

## 영상 모델에 실제로 넘어가는 것 (payloads.json `arms.r`)

- 이미지: 원본 시작+끝 프레임 2장 (B1과 같은 입력 형태 — 차이는 그림의 출신이 "원본"이라는 것뿐)
- 텍스트: B1과 동일한 동작 문장 + 3층 계약 (텍스트까지 동일해야 순수하게 이미지 계층만 분리됨)
- 요구 모델: B1과 동일 (시작+끝 동시 입력)

## 평가 시 주의

- 원본 프레임에는 **"DAAIKEEM" 워터마크**가 있다 — R 클립에 워터마크가 보여도 결함으로 세지 말 것.
- R보다 A~C가 잘 나오는 일은 구조상 없어야 정상이다. 만약 있다면 측정이나 생성 설정을 의심할 것.
- 판독 공식: `원본 ↔ R 격차 = 영상 모델의 한계` · `R ↔ 각 팔 격차 = 그 팔의 이미지 계층 손실`.
