# 실험 디렉토리 규칙 (2026-07-23 제정·시행 — 오너 승인)

> 근거가 된 고통: 결과 md 25개가 `results/`에 코드명(E9d…)으로 평평하게 쌓여 사람이 못 찾고,
> 한 실험의 문서·자산·스크립트가 `results/`·`assets/`·`tools/` 세 군데로 흩어지며,
> E번호가 의미가 아니라 파생 이력만 담게 됨 (E9→E9b→E9c→E9d-proto→E9d-copy).

## 규칙 1 — 폴더가 실험이다 (콜로케이션)

실험 하나 = 폴더 하나. 그 실험의 설계·결과·이미지·영상·스크립트·중간 데이터가 전부 그 폴더 안에 산다.
타입별(문서는 results/, 스크립트는 tools/)로 흩지 않는다.

## 규칙 2 — 이름은 사람 말, 코드번호 금지

- 폴더·파일명: **영문 케밥 슬러그** (한글 파일명은 mac↔linux 정규화 문제로 금지, 한글 제목은 문서 안에)
- 실험 폴더는 **ISO 날짜 접두**: `2026-07-23_input-format/`
- E번호는 오늘부로 폐기. 과거 번호는 INDEX의 "옛 이름" 열에만 남긴다(검색용 별칭).

## 규칙 3 — 2계층 고정: 트랙 / 실험

```
experiments/
  INDEX.md · CONVENTIONS.md · README.md   ← 루트의 md는 이 3개뿐
  utils/                        ← 공용층 — 오너가 들어갈 일 없는 것 전부 (아래 규칙 3-1)
    templates/                  ← design/result 템플릿
    tools/                      ← 트랙을 가로지르는 공용 스크립트
    campaign-2607/              ← 캠페인 메타 문서 (계획서·일괄판정·회고)
  <트랙-슬러그>/                 ← 트랙 = 진행형 주제 (예: continuity-copy)
    README.md                   ← 트랙 안내판: 무엇을 쫓는 주제인지, 하위 실험 목록·타임라인
    2026-07-23_<실험-슬러그>/    ← 실험 = 질문 1개
      design.md                 ← 설계 + 사전 판정 기준 (실행 전 확정)
      result.md                 ← 결과 (사람 언어 포맷 — 기존 작성 규칙 그대로)
      notes.md                  ← (선택) 진행 중 관찰 메모
      assets/                   ← 이미지·영상·json
      tools/                    ← 이 실험 전용 스크립트
```

- **규칙 3-1 (2026-07-23 오너 지시)**: experiments 루트에서 "들어가 보는 디렉토리"는 **트랙뿐**이어야
  한다 — 실험의 결과·가설을 보러 들어가는 곳. 템플릿·공용 스크립트·캠페인 메타 등 공용물은 전부
  `utils/` 아래로 (INDEX가 링크로 연결하므로 직접 브라우징할 필요 없음). 루트에 새 디렉토리를 만들
  때 "이게 트랙인가?"에 아니오면 utils행.
- **규칙 3-2**: 문서 안 이미지·영상은 **링크가 아니라 임베드**로 — `![|폭](경로)` (옵시디언 인라인
  렌더 호환). HTML `<img>` 태그와 일반 링크 `[라벨](x.jpg)`는 쓰지 않는다.
- **트랙** = 질문이 진화해도 유지되는 주제 단위 (예: "연출 연속성과 레퍼런스 카피").
- **실험** = 사전 판정 기준을 가진 질문 1개 + 그 측정. **질문이 바뀌면 폴더를 새로 판다**
  (지금까지 E9→E9b→E9c로 번호를 파던 것의 올바른 형태). 시제품·설계도도 실험 폴더 1개로 취급.
- 두 트랙에 걸치면: 주 트랙에 두고 INDEX에서 상호 링크. 3계층 이상 파지 않는다.

## 규칙 4 — 실험 폴더 안 고정 파일명

`design.md` / `result.md` / `notes.md` / `assets/` / `tools/` — 이름 고민 금지. 폴더가 이미 맥락이므로
파일명에 실험 이름을 반복하지 않는다. 자산은 역할 접두: `s01_start.jpg`, `filmstrip.jpg`, `final_60s.mp4`.

## 규칙 5 — 상태는 INDEX.md 한 곳

- INDEX.md = 트랙별 실험 목록 + 상태(⬜설계 / 🔵실행 / 🟡측정완료·판정대기 / ✅판정완료 / ⏸보류) +
  옛 E번호 별칭 + 한 줄 결론. **상태를 두 군데 적지 않는다** (지금 계획서 상태표가 INDEX로 승격).
- 오너 판정 기록은 해당 실험 `result.md` 머리말 + INDEX 상태 갱신. 별도 판정 문서는 여러 실험을
  한 번에 판정할 때만(트랙 루트에 `judgment_YYYY-MM-DD.md`).

## 규칙 6 — 문서 본문 작성 규칙 (기존 유지)

한 줄 결론 → 무엇을 의심했나 → 입력 원문 → 방법(사전 기준) → 출력 원문 → 결론 → 기술 부록.
본문은 사람 언어, 코드 식별자·모델명·env는 부록에만. (메모리 등재된 기존 규칙 — 변경 없음)

## 규칙 7 — 실행 흔적

- 하네스 로그(`logs/writer-stage-exp/`)는 위치 고정(하네스 제약). run-id는 `<실험슬러그 축약>-<팔><회차>`
  (예: `inpfmt-b1-2`). 실험 result.md 부록에 run-id 목록 필수.
- 대용량: 최종 산출 영상·대표 이미지·json 메타는 git 추적, 중간 클립·재생성 가능한 대량 파일은
  로컬만(assets/ 안에 두되 .gitignore) + result.md에 재생성 커맨드.

## 규칙 8 — research/ 최상위 배치: canon / records / references / experiments

2026-07-23 평탄화로 옛 `writer/` 중간 계층을 없애고 `research/` 최상위를 4개로 나눴다:

```
research/
  canon/          ← 살아있는 정본 (무날짜 이름). 예: writer-job-model.md
  records/        ← 시점 기록 (날짜 접두 유지). 감사·세션 노트
  references/     ← 외부 레퍼런스 해석 + assets (슬러그 규칙 공유)
  experiments/    ← 이 문서가 규율하는 실험 트랙들
```

- **canon/** = 살아있는 정본. 무날짜 이름, **항상 현재 믿는 최신 상태**를 유지한다. 개정은
  실측(실험 판정) 근거로만 하고, 개정 이력은 git과 본문 판정 기록이 담당한다.
- **records/** = 시점 기록. 특정 시점의 감사·세션 스냅샷이라 **날짜 접두를 유지**하고 사후 개정하지 않는다.
- **검증 끝난 지식은 실험 트랙에서 canon으로 승격**한다 (실험 폴더는 근거, canon은 현재 결론).
- 레퍼런스 해석은 `references/`, 그 레퍼런스를 쓰는 실험은 트랙 폴더 — 서로 링크.
